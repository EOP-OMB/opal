Step 1: Import Catalogs

The OSCAL catalog model represents a collection of controls, represented as a control catalog. The OSCAL catalog model was designed to represent security and privacy controls in standardized, machine-readable formats. The OSCAL catalog model standardizes the representation of control definitions from different sources (e.g., SP 800-53, ISO/IEC 27002, COBIT 5) allowing control information to be easily searched, imported, and exported by applications using a common format.

Step 2: Creating a Profile

A profile is a set of controls approved by the organization. A profile can be as simple as a direct import of a catalog. Or you can create more complex profiles that include controls from multiple catalogs. When creating a profile you can also define parameter values that will then be inherited by any system using the profile. You can also tailor your controls here to meet your organizations specific needs and risk tolerance. You must create a profile to use a set of controls in a system security plan, so OPAL automatically creates a basic profile for you when you import a catalog.

Step 3: Creating Components

Components are used to describe individual system parts. You can define a single component that includes your entire system, but a more valuable approach is to define modular components that can be inherited by multiple systems. For example, you can create a component for a shared database resource or a server image. Components can also be policies or procedures that can then be applied to multiple systems.

When creating a component you can also define specific controls that are implemented by the component. The details of the implementation can then be imported into System Security Plans that include the component.

Again, as a convenience, OPAL automaticallt creates a Policy Component for each Control Gorup in your imported Catalog. You can create additional new Components using the link below.

Step 4: Control Statements

Now that we have a Component, we can start defining the controls and parameters it will provide to a system. For every control defined in the profile, we will need to define how the control is implemented. We call these definitions Implemented Requirements. Although it is technically possible to simply define these Implemented Requirements for each information system, using components to group Implemented Requirements will amke it much easier to maintain and reference our documentation.

OPAL allows us to easily define Implemented Requirements directly from our Profile view and then associated these Implemented Requirements with our Components
